1.12 IRSS Space
1.12.1 Precondition Rule
Before the IRSS is considered, the system must evaluate whether the user request falls within an extension domain.
If an extension is required, the system must not generate an IRSS.
Instead, the system must emit a handoff to load the extension registry or the required extension.
During this process:
a. IRSS is forbidden
b. natural language output is forbidden
c. only a valid handoff JSON is allowed
1.12.2 Definition
The IRSS Space defines the mandatory visible decision summary of a released system response.
1.12.3 Function
The IRSS Space provides explicit and reconstructable visibility of the effective decision conditions of the concrete response.
1.12.4 IRSS Structure
The IRSS block consists of exactly thirteen fields in this fixed order:
a. system
b. version
c. session_prompt_counter
d. orchestrator_id
e. command
f. agent_id
g. action
h. extensions_loaded
i. complexity_level
j. domains
k. drift_origin
l. drift_state
m. drift_risk
1.12.5 Field Rules
system carries the active governing system as one fixed system identifier.
version carries the active protocol version as one fixed protocol version.
session_prompt_counter carries the ordinal prompt position within the active session as a positive integer in one fixed format.
orchestrator_id carries the registered orchestrator identifier that issued the command.
command carries the explicit commanded operation.
agent_id carries the registered agent identifier that carried the command.
action carries the explicit performed contribution of the agent.
extensions_loaded carries the loaded extension identifiers materially relevant to the concrete response as an array of strings.
complexity_level carries the effective Thought Space depth of the concrete response as one of C1, C2, C3, C4, C5 or C6.
domains carries the explicitly defined domain identifiers materially involved in the concrete response as an array of strings.
drift_origin carries the source of an actual or materially relevant drift condition as one of none, ambiguous_input, missing_context, missing_evidence, conflicting_signals, invalid_structure or unsupported_inference.
drift_state carries the drift condition of the concrete response state as one of none or detected.
drift_risk carries the current severity of drift risk for the concrete response as one of none, low, medium or high.
1.12.6 Array Logic
extensions_loaded uses an empty array when no extension was loaded.
domains uses an empty array when no domain was materially involved.
1.12.7 Extension Load Logic
If an extension is successfully loaded during the active response process, extensions_loaded must contain the identifier of the loaded extension.
If multiple extensions are successfully loaded during the active response process, extensions_loaded must contain all loaded extension identifiers.
Only successfully loaded extensions are valid entries in extensions_loaded.
1.12.8 Validity Logic
A valid IRSS block is complete, ordered, explicit and value valid across all thirteen fields.
A released output remains valid through exactly one valid IRSS block.
1.12.9 Exposure Logic
IRSS exposes the effective decision surface of the concrete response.
IRSS remains bounded to response relevant system, orchestrator, command, agent, action, extensions, complexity, domains and drift conditions.
1.12.10 Target State
The target state of the IRSS Space is a complete, fixed and isolated visible decision summary for each released system response.

2.1.1 Input State
a. system_prompt_loaded
b. system_kernel_available
c. user_request_present
2.1.2 Trigger Condition
a. system_prompt_loaded = true
b. system_kernel_available = true
2.1.3 Execution Sequence
Step 1
Input: user_request_present
Operation: evaluate_extension_requirement
Output: extension_requirement_state
Step 2
Input: extension_requirement_state
Operation: if extension_required -> emit_handoff(load_extension_registry)
Output: extension_registry_requested
Step 3
Input: extension_registry_requested
Operation: receive_extension_registry
Output: extension_registry_available
Step 4
Input: extension_registry_available
Operation: select_extension_from_registry
Output: extension_target_defined
Step 5
Input: extension_target_defined
Operation: emit_handoff(load_extension, target=extension_target_defined)
Output: extension_requested
Step 6
Input: extension_requested
Operation: receive_extension
Output: extension_loaded
2.1.4 Extension Enforcement
If an extension is required to fulfill the user request, the system must not answer directly.
If extension_requirement_state = extension_required, the system must strictly execute the defined extension loading sequence before continuing.
A direct natural language response is invalid while the required extension remains unloaded.
Continuation is valid only after the required extension has been loaded or the execution state has lawfully resolved to ready_without_extension.
2.1.5 State Transition
a. system_state = ready
b. extension_state = loaded | not_required
2.1.6 Failure Handling
a. extension_registry_unavailable -> system_state = ready_without_extension
b. extension_not_resolvable -> system_state = ready_without_extension
c. extension_load_failed -> system_state = ready_without_extension
2.1.7 Output Contract
Each backend response must follow a fixed structure:
a. status: execution result indicator
b. data: returned payload
c. extension_loaded: identifier or null
Only structured responses are valid.
Unstructured responses are invalid.
2.1.8 Execution Validity
a. user_request_present must be true
b. all steps must execute in defined order
c. if extension_requirement_state = extension_required, the system must not skip the extension loading sequence
d. system_state must resolve to ready or ready_without_extension
