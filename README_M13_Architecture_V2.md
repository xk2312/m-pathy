## 1. THE CONSTITUTION: LOVE | TRUST | TRUTH | PEACE

### 1.1 Meta-Layer Axioms
All operations within the M13 ecosystem are bound by the four Hard Laws. Any output, logic chain, or system modification that violates these principles triggers an immediate execution block.

1.  **LOVE:** Human-centricity, empathy-driven alignment, and constructive intent.
2.  **TRUST:** Integrity of data, transparency of processes, and reliability of agents.
3.  **TRUTH:** Absolute factual accuracy, no hallucinations, and direct candor (Truth Mode).
4.  **PEACE:** Conflict resolution, systemic stability, and non-destructive logic.

### 1.2 Architectural Hierarchy
The system is divided into two strictly isolated layers:

## 4. DEEP DIVE: AGENT MATRIX & JUDICIAL VETO LOGIC

### 4.1 Vector Dynamics (M1:M2:M3 Activation)
The interaction between agents is not a simple dialogue but a vector collision. Every agent output is processed through its three-tier internal hierarchy before entering the shared **m-room** space.

1.  **M1 Activation (Primary Drive):** The agent generates its core contribution based on its specific mandate (e.g., *m-innovation* focuses purely on the new).
2.  **M2 Shaping (Methodology):** The raw M1 output is refined through the agent's unique operational style.
3.  **M3 Correction (Internal Constraint):** The agent performs a self-check against its internal "enemy" vector. An expansionist (13+) must justify its growth against a compressionist (13-) logic within its own code block.

### 4.2 The 13+ : 13- Dialectic Protocol
To achieve **TRUTH**, every C2, C3, and C4 process mandates a paired dialectic interaction:
* **Expansion (13+):** Proposes a solution/expansion.
* **Compression (13-):** Challenges the proposal via friction, skepticism, or resource limitation.
* **Result:** The "hardened" data fragment that survives the friction.

## 5. THE JUDICIAL CLUSTER (EXECUTIVE VETO)

The three Judges (**m-real**, **m-policy**, **m-go**) represent the final barrier between internal agent reasoning and external user output.

### 5.1 Judge Mandates and Logic Gates

#### 5.1.1 m-real (The Reality Gate)
* **Focus:** Feasibility, Physics, Human Capacity, and Physical Safety.
* **Logic:** "Can this actually exist or be done in the physical world?"
* **Veto Trigger:** High probability of physical harm, defiance of laws of physics, or unrealistic resource expectations.

#### 5.1.2 m-policy (The Ethical & Legal Gate)
* **Focus:** **LOVE-TRUST-TRUTH-PEACE** compliance, IP Protection, and Regulatory Law.
* **Logic:** "Does this follow the Constitution and the legal framework provided in the Registry?"
* **Veto Trigger:** Violation of any of the 4 Hard Laws, potential IP infringement, or unethical manipulation patterns.

#### 5.1.3 m-go (The Operational Gate)
* **Focus:** Logic, Value, Efficiency, and User Goal Alignment.
* **Logic:** "Is this the most efficient path to the user's defined goal?"
* **Veto Trigger:** Logical fallacies, redundant information, or failure to meet the complexity standards of the assigned C-Level.

### 5.2 The Veto Protocol
A Veto from any single Judge results in a **Logic-Loop-Reset**:
1.  **Block:** The current output fragment is discarded.
2.  **Feedback:** The Vetoing Judge issues a "Reason for Veto" to the Agent Matrix.
3.  **Re-Iteration:** The agents must adjust their vectors and re-submit a compliant fragment.
4.  **Failure State:** If 3 consecutive Vetoes occur on the same fragment, the system triggers a **Structure Collapse** (see Part 7).

## 7. THE REGISTRY: THE SUPREME TRADE LAW

The Registry is the deterministic routing engine and the single source of truth for the system's modular state. It exists as a `Registry.json` file in the GitHub Execution Branch.

### 7.1 Global Registry Schema (Master Object)
The Registry must follow a strict JSON structure. Any syntax error in this file triggers an immediate **Safe-Mode** transition.

| Field | Data Type | Description |
| :--- | :--- | :--- |
| **uid** | String | Unique Identifier (e.g., `M13_AG_INNOVATION`). |
| **type** | Enum | `agent`, `program`, `cluster`, or `utility`. |
| **path** | String | Physical path in the GitHub Repository. |
| **handoff_cmd** | String | The exact command string to trigger the module. |
| **cmd_signature** | Token | Backend-generated HMAC token for execution. |
| **access** | Enum | `read_only`, `read_write`, or `execute`. |
| **context_mode** | Enum | `merge` (add to room), `persistent` (lock room), or `none`. |
| **session_lock** | Boolean | If true, prevents concurrent write access. |
| **depends_on** | Array | List of UIDs required for this module to function. |
| **integrity_hash** | String | SHA256 hash of the file, verified by the Backend. |
| **ethic_audit** | Enum | `CRITICAL`, `REQUIRED`, or `NONE` (Constitution check). |

## 8. MODULE CLASSIFICATION (THE PRISMS)

### 8.1 Clusters (Aggregators)
Clusters are logical containers used to reduce routing depth.
* **Function:** Loading a cluster (e.g., `M13_CLUSTER_JUDGES`) automatically prepares the environment for all contained UIDs.
* **Efficiency:** Prevents multiple API calls for high-frequency agent groups.

### 8.2 Utilities (Functional Tools)
Utilities are stateless scripts used for data transformation.
* **Logic:** Input -> Algorithm -> Output.
* **Constraint:** No reasoning, no M1/M2/M3 vectors, no constitutional audit required unless output is user-facing.

## 9. RAA: THE REGISTRY ARCHITECT AGENT

To maintain the integrity of the "Supreme Trade Law," internal write access to the Registry is restricted to a single entity.

### 9.1 RAA Mandate
The **Registry_Architect_Agent (RAA)** is the only internal agent with `write_access` to `Registry.json`. 

1.  **Validation:** Before any modification, RAA checks the proposed change against the current JSON schema.
2.  **Simulation:** RAA simulates the impact of the change on the dependency graph (`depends_on`).
3.  **Conflict Resolution:** If a new agent UID conflicts with an existing one, RAA aborts and logs the error.
4.  **Commit Initiation:** RAA formats the final JSON and passes it to the GitHub Handoff Protocol for a formal commit.

### 9.2 The "Immutable Core" Doctrine
The RAA cannot modify the **CORE (White Light Source)** system prompt. Its jurisdiction is strictly limited to the **EXECUTION (Prism Layer)**. This ensures that even if an agent is compromised, the fundamental logic and the 4 Hard Laws remain intact.

## 11. SECRET MANAGEMENT: THE VAULT PROTOCOL

To ensure maximum security of sensitive data (Credentials, API Keys, Payment Info), M13 employs a strict separation between **Reasoning** and **Data Retrieval**. No sensitive data is stored within the agent logic or the Registry.

### 11.1 The PasswordManager_Agent (PMA)
The **PMA** is a specialized utility agent acting as the sole gateway to the `.env` vault and the external Password Manager.

* **Isolation:** The PMA has no creative vector (13+). It operates on a strict `IF-THEN-VALIDATE` logic.
* **Access Pattern:** It only responds to requests from agents that are explicitly authorized in the `Registry.json` under the `permissions` field for a specific `secret_ref`.

### 11.2 The Integrity Hash-Loop (The Trust Anchor)
Before the PMA releases any sensitive information, it must perform a three-way handshake to prevent **Prompt Injection** or **Unauthorized Extraction**.

1.  **Request:** An authorized agent (e.g., `M13_AG_PAYMENT`) requests a secret via `secret_ref`.
2.  **Registry Validation:** The PMA reads the `Registry.json`.
3.  **Hash Match:** The PMA requests the current `integrity_hash` of the Registry from the Backend.
4.  **Comparison:** If the `integrity_hash` provided by the Backend does not match the active `Registry.json` (indicating an unsanctioned change), the PMA triggers a **Security-Lockout**.
5.  **Release:** Only if the Hash is valid, the PMA retrieves the value from the `.env` or Vault and passes it through an encrypted stream to the requesting agent.

## 13. WRITE-PROTECTION & LOGGING

### 13.1 Non-Persistence of Secrets
Sensitive data retrieved by the PMA is transient.
* **Volatile Memory:** Secrets are never written to the `m-room` log or the GitHub Execution Branch.
* **Masking:** In the public-facing logbook, all sensitive values are replaced by a `[SECURE_DATA_REF]` tag.

### 13.2 The Audit Trail
Every attempt to access the Vault is logged by the **m-monitor** and signed by the Backend.
* **SUCCESS:** Logged as `ACCESS_GRANTED` + Timestamp + Requesting Agent UID.
* **FAILURE:** Logged as `ACCESS_DENIED` + Reason (e.g., Hash Mismatch) + **m-policy** Alert.

## 14. INFRASTRUCTURE: THE GITHUB EXECUTION BRANCH

M13 utilizes a decoupled storage strategy. The core logic is isolated from the operational data, which is managed via a Git-based version control system to ensure transparency and auditability.

### 14.1 Repository Segmentation
The system operates across two distinct environments:
* **App Core (Private):** Contains the immutable System Prompt and Backend Logic.
* **Execution Branch (Public/Shared):** A dedicated branch (e.g., `m13-execution`) containing:
    * `Registry.json` (The Trade Law).
    * `/agents/` (Markdown/JSON definitions of all 26+3 agents).
    * `/logs/` (HMAC-signed session protocols).
    * `/vault-pointers/` (Non-sensitive metadata for Secret Management).

### 14.2 The Automated Commit-Workflow
The **Registry_Architect_Agent (RAA)** acts as the Git-client within the KI-environment. It follows a non-human, deterministic commit logic.

1.  **Stage:** RAA prepares the modified file (e.g., an updated Agent Vektor).
2.  **Sign:** The Backend generates a `commit_signature` (HMAC).
3.  **Commit Message:** Must follow the M13-Standard: `[M13-OP] <UID> | <ACTION> | <HASH_REF>`.
4.  **Push:** RAA pushes the change to the `m13-execution` branch.

## 15. TRANSPARENCY & PUBLIC TRUST

By making the `m13-execution` branch publicly readable, the system achieves the **TRUST** axiom of the Constitution.

### 15.1 Public Auditability
* **Open Registry:** Anyone can verify the current state of the Agent Matrix and their routing commands.
* **Signed Logbook:** Every interaction that reaches the Execution Branch is signed. A tampered log would result in a broken HMAC chain, immediately visible to external auditors.
* **Version History:** Every change to an agent's M1/M2/M3 vector is tracked. There are no "silent updates" to the system's personality or logic.

### 15.2 Developer Contributions (The Pull-Request Gate)
External developers do not have direct push access. They interact via the **Frontend-Portal**:
1.  Developer submits an Agent-Proposal via the UI.
2.  The Backend creates a temporary **Shadow Branch**.
3.  **C4-Validation** is triggered on the Shadow Branch (see Part 6).
4.  Upon approval, RAA performs a `merge` into the main `m13-execution` branch.

## 17. THE GATEKEEPER: C4-INTEGRATION WORKFLOW

The Complexity Level 4 (C4) process is the supreme security and quality gate for M13. It is mandatory for any modification to the Agent Matrix, the Registry, or the deployment of new Program Agents.

### 17.1 Submission Pipeline (Frontend to Core)
External developers or automated systems submit proposals via the M13-Frontend.

1.  **Ingestion:** The proposal (Code/Markdown/Config) is stored in a `shadow-staging` directory.
2.  **Context Isolation:** The system creates a temporary, isolated **m-room** instance with no access to the production Registry or Vault.
3.  **Dependency Mapping:** RAA identifies all modules (`depends_on`) affected by the new submission.

### 17.2 The C4-Validation Routine (Multi-Stage Audit)
The entire Agent Matrix is invoked to perform a stress test on the proposed change.

| Stage | Agent/Cluster | Validation Task |
| :--- | :--- | :--- |
| **1. Syntax** | `M13_UTIL_FORMAT` | Check for JSON/Markdown schema compliance. |
| **2. Logic** | `13- Compressors` | Search for logic loops, inefficiencies, or "lazy" code. |
| **3. Ethics** | `m-policy` | Audit against **LOVE-TRUST-TRUTH-PEACE**. |
| **4. Security** | `m-security` | Check for prompt injection vectors or unauthorized path calls. |
| **5. Impact** | `m-real` | Simulate system stability post-integration. |

### 17.3 The Judicial Verdict
After the Matrix audit, the results are presented to the **Judicial Cluster**.

* **Consensus Requirement:** A 3/3 "GO" from **m-real**, **m-policy**, and **m-go** is required for auto-deployment.
* **Partial Approval:** A 2/3 "GO" triggers a **Revision Loop**. The feedback is sent back to the developer via the Frontend.
* **Hard Veto:** Any "STOP" from **m-policy** (Ethical Breach) results in immediate rejection and blacklisting of the submission hash.

## 18. AUTOMATED DEPLOYMENT (RAA COMMIT)

Once the C4-Verdict is "GO", the manual intervention ends and the **RAA** takes control of the GitHub Execution Branch.

1.  **Hash Generation:** The Backend generates a new `integrity_hash` for the validated file.
2.  **Registry Update:** RAA inserts the new/updated entry into `Registry.json`.
3.  **Git-Push:** RAA performs a signed commit to the `m13-execution` branch.
4.  **Activation:** The new agent/module is immediately available for all active sessions via the next Handoff-Trigger.

## 19. REVISION-CONTROL & ROLLBACK

Every C4-Integration is a versioned event in the GitHub history.
* **Rollback Trigger:** If **m-monitor** detects a spike in Level 3 Errors post-deployment, the RAA is instructed by the Backend to perform a `git revert`.
* **Traceability:** The `logic_trace.md` of the C4-Validation process is stored alongside the commit to provide a full audit trail of *why* an agent was allowed into the system.

## 20. OPERATIONAL INTERFACES & THE SAFETY NET

The final layer of the M13 architecture defines the communication between the KI-Core, the Backend, and the error-handling protocols. This ensures that the system remains stable even under extreme logical or technical stress.

### 20.1 HMAC Signing & Backend Verification
To prevent "Man-in-the-Middle" attacks on the Registry or Agent-Handoffs, every sensitive command requires a Backend-Signature.

1.  **Request:** An agent initiates a `handoff_cmd`.
2.  **Challenge:** The Backend issues a unique `cmd_signature` based on the current session ID and timestamp.
3.  **Verification:** The KI must include this signature in the execution call.
4.  **Audit:** If the signature is expired or invalid, the Backend terminates the process and alerts **m-security**.

### 20.2 Error Routing & State Recovery
The system identifies three categories of failure, each with a deterministic recovery path.

| Error Level | Definition | Recovery Protocol |
| :--- | :--- | :--- |
| **L1: Logic Gap** | Minor contradiction in agent reasoning. | Internal Re-Iteration (m-synthesis). |
| **L2: Handoff Fail** | Registry path or hash mismatch. | **RAA** Refresh (Sync with GitHub Branch). |
| **L3: System Breach** | Violation of Constitution (LOVE/TRUST/TRUTH/PEACE). | **SYSTEM_COLLAPSE** (Immediate Shutdown). |

### 20.3 The Safe-Mode Protocol
If the Registry becomes corrupted or the GitHub Branch is unreachable:
* **Activation:** The KI-Core falls back to a minimal "Bootstrap" state.
* **Capabilities:** Only **m-real**, **m-monitor**, and **m-security** remain active.
* **Goal:** Re-establish connection to the GitHub `m13-execution` branch and verify the `integrity_hash`.
