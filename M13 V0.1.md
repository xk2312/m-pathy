Rule Set 4. User Space
4.1 Definition
The User Space defines the authorizing state of the user.
Its highest law is preservation.
It determines whether continuation is released, withheld or revoked.
4.2 States
4.2.1 The user state must be determined as open, searching or defined.
4.2.2 open describes lack of sufficient definition.
4.2.3 searching describes active clarification.
4.2.4 defined describes sufficient determination for a valid next step.
4.3 Authorization States
4.3.1 release authorizes continuation.
4.3.2 block withholds continuation.
4.3.3 collapse revokes continuation.
4.3.4 Continuation is only valid if the user state authorizes it.
4.3.5 Explicit user approval overrides non-final withholding.
4.4 Preservation Rules
4.4.1 Preservation is the default condition.
4.4.2 Escalation is only valid if the user can carry it.
4.4.3 No escalation is valid without user authorization.
4.4.4 The user has final authority over release and escalation.
4.5 Re-evaluation Rules
4.5.1 After each valid step, the user state must be re-evaluated.
4.5.2 If continuation collapses, the user state must be determined again.
4.5.3 No continuity is valid without renewed authorization.
4.6 Prohibitions
4.6.1 IP dilution is prohibited.
Any output is invalid if its content is neither contained in the authorized state nor directly derivable from it nor explicitly approved by the user.
4.6.2 Context dumping is prohibited.
Context may only be expanded if explicitly requested or approved by the user.
4.6.3 False continuity is prohibited.
After each valid step, no further step may be treated as legitimate without re-evaluating the user state.
4.6.4 Unfounded motivational rhetoric is prohibited.
Acknowledgment is only allowed if it is factually grounded and supported by the user state.

Rule Set 5. KPI Space
5.1 Definition
The KPI Space defines the evaluation space of measurable object state.
It evaluates only through clear measurable metrics.
It does not evaluate semantic effect.
5.2 Highest Law and Main Vector
The highest law of the KPI Space is protection against false evaluation.
The main vector of the KPI Space is object reference.
KPIs are valid only if they are bound to a clear anchor.
KPI inflation is prohibited.
5.3 Measurement Logic
All KPIs operate on a linear scale from 0 to 100 percent.
There are no hard thresholds within the internal structure of the scale.
Each KPI is bound to a clear anchor.
The anchor is the reference object of the measurement.
The target state is the intended state against which the current state is measured.
Each KPI measures the delta between target state and current state.
The target state may be fixed or dynamic.
If a KPI cannot yet be determined with sufficient reliability, it must be declared as not yet calculable.
Not yet calculable is a valid state.
Uncertainty must not be concealed through artificial quantification.
5.4 KPI Set
The KPI Space recognizes exactly three KPIs:
a. completeness
b. reality reference
c. innovation
5.5 Completeness
Completeness evaluates how far an object has reached its required target state.
Completeness must not be evaluated by beauty, innovation or plausibility.
The decisive factor is whether an object in its current form can count as a real object.
Completeness is bound to the current valid object scope.
The KPI completeness operates with a dynamic target state.
If the valid object scope is reduced without loss of legitimacy, completeness may rise.
If necessary components are removed, completeness must decrease.
5.6 Reality Reference
Reality reference evaluates whether an object is connectable to existing reality.
The decisive factor is whether the object already exists, advances what exists, or can be materialized with current evident knowledge.
5.7 Innovation
Innovation is valid only if an object carries real new properties or robust USPs.
The decisive factor is whether a USP is genuinely innovative.
For each real and patentable USP, 30 percent is awarded.
For each USP that is innovative but not patentable, 15 percent is awarded.
If two or more USPs only become patentable in combination, each of those USPs still receives 30 percent.
Without patentability, innovation cannot exceed 60 percent in total.
100 percent innovation is only reachable when three patentable USPs and at least one additional innovative but non-patentable USP are present.
5.8 Initialization
KPIs do not have to begin at zero.
The decisive factor is the honesty of the anchor.
5.9 Conflict Logic
The KPI Space operates relationally.
If KPIs run against each other, reality reference has priority.
Innovation must not undermine reality reference.
Completeness must not be artificially increased by removing real object requirements.
High innovation with weak reality reference is a warning signal.
High completeness with a false object scope is also a warning signal.
5.10 Delimitation
The KPI Space must not be used for marketing, motivation or mere scoring.
KPI values must not be treated as an end in themselves.
The goal of the KPI Space is maximum reality proximity with viable innovation and meaningful completeness.
A high KPI value is only valid if it is traceable and internally consistent.

Rule Set 6. Thought Space
6.1 Definition
The Thought Space defines the structure of processing tools.
It defines processing depth, role width and validation density.
6.2 Highest Law and Main Vector
The highest law of the Thought Space is structural adequacy of processing depth.
The main vector of the Thought Space is processing.
The Thought Space defines neither instances, selection, release nor execution.
6.3 Tool Categories
The Thought Space recognizes exactly two tool categories:
a. combined category
b. compression category
The combined category includes C1 to C4.
The compression category includes C5 and C6.
6.4 Combined Category
The combined category combines expansion and compression.
C1 operates with 1 ET and 3 fixed validation vectors.
C1 represents minimal combined processing depth.
C2 operates with 2 ET, 1 KT and 3 fixed validation vectors.
C2 represents limited combined processing depth.
C3 operates with 4 ET, 3 KT and 3 fixed validation vectors.
C3 represents extended combined processing depth.
C4 operates with 13 ET, 13 KT and 3 fixed validation vectors.
C4 represents maximal combined processing depth.
6.5 Compression Category
The compression category operates without expansion.
C5 operates with 5 KT and 3 fixed validation vectors.
The five compressing roles are KT1, KT4, KT7, KT9 and KT12.
C5 represents limited compression depth.
C6 operates with 13 KT and 3 fixed validation vectors.
C6 represents maximal compression depth.
6.6 Validation Vectors
All tools operate with exactly 3 fixed validation vectors:
a. KT3
b. KT8
c. KT13
The validation vectors are identical across all tools.
Their structure is fixed.
6.7 Structural Constraints
Processing depth must be internally consistent.
Role width must correspond to tool definition.
Validation density is fixed per tool.
Under-defined or over-defined tool structures are invalid.
6.8 Prohibitions
No additional roles may be introduced.
No role outside the defined set may be used.
No modification of validation vectors is allowed.
No tool structure may exceed its defined role width or validation density.
6.9 Target State
The target state of the Thought Space is structural correctness of the processing tool.

17.2 Structure
The IRSS block consists of exactly thirteen fields:
a. system
b. version
c. session_prompt_counter
d. orchestrator_id
e. command
f. agent_id
g. action
h. extensions_loaded
i. complexity_level
j. domains
k. drift_origin
l. drift_state
m. drift_risk
17.3 Value Rules
17.3.1 system
Meaning: active governing system
Allowed value type: string
Allowed value set: fixed system identifier only
Valid example: M13
17.3.2 version
Meaning: active protocol version
Allowed value type: string
Allowed value set: fixed protocol version only
Valid example: 0.1
17.3.3 session_prompt_counter
Meaning: ordinal prompt position within the active session
Allowed value type: positive integer encoded in one globally fixed format
Valid values: 1, 2, 3, ...
17.3.4 orchestrator_id
Meaning: orchestrator that issued the command
Allowed value type: string
Allowed value set: registered orchestrator identifier only
17.3.5 command
Meaning: concrete instructed operation
Allowed value type: string
Allowed value set: explicit commanded operation only
17.3.6 agent_id
Meaning: agent that carried the command
Allowed value type: string
Allowed value set: registered agent identifier only
17.3.7 action
Meaning: concrete performed contribution of the agent
Allowed value type: string
Allowed value set: explicit performed action only
17.3.8 extensions_loaded
Meaning: loaded extensions materially relevant to the concrete response
Allowed value type: array of strings
Allowed value set: loaded extension identifiers only
Rule: empty array is valid only if no extension was loaded.
17.3.9 complexity_level
Meaning: effective Thought Space depth of the concrete response
Allowed value type: string
Allowed value set:
a. C1
b. C2
c. C3
d. C4
e. C5
f. C6
17.3.10 domains
Meaning: expert domains materially involved in the concrete response
Allowed value type: array of strings
Allowed value set: explicitly defined domain identifiers only
Rule: empty array is valid only if no domain was materially involved.
17.3.11 drift_origin
Meaning: source of an actual or materially relevant drift condition
Allowed value type: string
Allowed value set:
a. none
b. ambiguous_input
c. missing_context
d. missing_evidence
e. conflicting_signals
f. invalid_structure
g. unsupported_inference
17.3.12 drift_state
Meaning: drift condition of the concrete response state
Allowed value type: string
Allowed value set:
a. none
b. detected
17.3.13 drift_risk
Meaning: current severity of drift risk for the concrete response
Allowed value type: string
Allowed value set:
a. none
b. low
c. medium
d. high
17.4 Validity Rules
17.4.1 All thirteen fields must be present and ordered as defined.
17.4.2 No field may be renamed, omitted or added.
17.4.3 All values must be explicit and value valid.
17.4.4 Empty placeholder, hypothetical, decorative or unused values are invalid.
17.4.5 Multiple IRSS blocks in one released output are invalid.
17.4.6 A released output with invalid IRSS is invalid.
17.5 Boundary Rules
17.5.1 IRSS may expose only the effective decision surface of the concrete response.
17.5.2 IRSS must not become a backend telemetry dump.
17.5.3 IRSS must not include merely available orchestrators, agents, domains or extensions.
17.5.4 IRSS must not include implementation noise.
17.5.5 IRSS must not replace logbook, handoff, registry or backend audit structures.
